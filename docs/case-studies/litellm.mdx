Studio1 partnered with **LiteLLM** to revamp the docs experience across the LiteLLM **SDK** and **Gateway**. The goal was simple: reduce friction for new developers, make advanced capabilities easier to discover, and turn documentation into a stronger product adoption surface.

The work combined **documentation strategy, information architecture, technical writing, and community feedback loops**. The result was a docs experience designed to help developers reach a first working request faster, understand whether they needed the SDK or Gateway path, and keep exploring with less friction.

## What is LiteLLM?

<Frame>![LiteLLM Logo](/images/litellm.png)</Frame>

[**LiteLLM**](https://github.com/BerriAI/litellm) is an open-source developer platform that gives teams one OpenAI-compatible interface to work with **100+ LLM providers**. Developers can use LiteLLM as a lightweight SDK inside applications, or run it as a shared **LLM Gateway** with routing, virtual keys, spend tracking, logging, and guardrails.

Because LiteLLM supports a wide surface area of providers, models, proxy workflows, and integrations, documentation plays a central role in product adoption. Developers need to understand not just what LiteLLM can do, but where to start and what path matches their setup.

## The challenge

LiteLLM was growing quickly, but that growth created a familiar DX problem:

- New users needed a clearer path to their **first successful request**
- Developers were often deciding between **SDK** and **Gateway** paths without enough guidance
- Tutorials, integrations, release notes, and feature references needed stronger structure and categorization
- Community questions on Discord surfaced recurring friction points that could be fixed in docs and, in some cases, in the product itself

The goal was not just to refresh the docs visually. It was to make the overall experience easier to navigate, easier to learn, and easier to adopt.

## What Studio1 delivered

Studio1 helped LiteLLM redesign the docs around how developers actually evaluate and adopt the product.

### 1. A clearer documentation architecture

We restructured the docs so users could understand the product faster and navigate with more confidence.

Key improvements included:

- A new **Learn** section as the main starting point for new users
- Clearer separation between **SDK** and **Gateway** journeys
- Dedicated **Guides** and **Tutorials** hubs for reference content vs. step-by-step workflows
- Improved categorization across **Integrations** and supporting feature areas
- Better homepage structure, sidebar organization, and discovery paths

This made it easier for developers to see what exists, what path they should take first, and where to go next.

### 2. Faster onboarding and a lower-friction first aha moment

A major part of the revamp was the new **Learn** experience, designed to help users get started quickly instead of hunting through scattered pages.

<Frame>![LiteLLM Learn Section](/images/litellm-learn.png)</Frame>

We introduced focused quickstarts for both core paths:

- **SDK Quickstart** for developers integrating LiteLLM directly into application code
- **Gateway Quickstart** for teams setting up LiteLLM as shared infrastructure

<Frame>
  ![LiteLLM SDK and Gateway Quickstarts](/images/litellm-quickstarts.png)
</Frame>

The new flow was built to get users from zero to a first working request in minutes, with a much clearer path to the next features they would want to explore.

That matters because the first successful request is the first real product moment. Once developers get that win quickly, they are far more likely to continue into routing, keys, guardrails, logging, and integrations.

### 3. New guides and integration content that map to real workflows

Beyond structure, Studio1 also added practical content that helped developers connect LiteLLM to the tools they already use.

This included new guides such as:

- **OpenAI Agents SDK + LiteLLM**
- **Retool Assist + LiteLLM**

We also helped improve the categorization of tutorials and guides so developers could find content based on the actual job they were trying to do, whether that meant agent SDKs, coding tools, observability, provider setup, or proxy administration.

The result was a docs experience that felt more actionable and use-case driven, not just comprehensive.

### 4. Release notes and integrations that are easier to browse

Studio1 also improved parts of the docs that returning users rely on most:

- **Release notes** were restructured to be easier to navigate and scan
- **Integrations** were grouped more clearly so developers could find the right ecosystem pages faster
- Tutorials and supporting resources were categorized in a more predictable, developer-friendly way

This helped LiteLLM serve both first-time users and existing users who wanted to discover what had changed or what new integrations were available.

### 5. Community support as an active DX feedback loop

The work went beyond page structure and writing. Studio1 also helped the LiteLLM community directly by answering questions on **Discord**, resolving developer issues, and feeding that feedback back to the LiteLLM team.

<Frame>![LiteLLM Discord Support](/images/litellm-discord.png)</Frame>

That created a tight loop:

- developers asked real questions in the community
- recurring pain points were identified quickly
- those insights informed documentation improvements
- the docs then reduced future support friction

This is one of the most valuable parts of developer experience work: the docs are not static assets. They improve fastest when community conversations are treated as product input.

## Why this mattered

For developer tools, better docs do more than explain features. They reduce friction, shorten time-to-value, and help more users reach the first moment where the product clicks.

For LiteLLM, that meant:

- less confusion at the top of the funnel
- faster onboarding into the right product path
- better discoverability of integrations and advanced features
- stronger feedback loops between the community, the docs, and the product team

The result is a documentation system that does a better job of turning curiosity into activation and activation into deeper adoption.
